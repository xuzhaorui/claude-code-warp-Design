#001 | 2026-05-12 | 回滚方式错误：手动编辑代码代替 git 回滚

【问题】
用户要求撤销上一次 UI 改动（Hero 卡片扫码图标区域）。
我选择手动编辑代码还原，而不是用 git 命令回滚。
违反了 CLAUDE.md 编码纪律："Git 是后悔药：每次变更需当前分支本地提交"。

【根因】
- 没有在每次变更后立即 commit，导致没有可回滚的 commit 节点
- 习惯性用代码编辑思维解决问题，而不是用版本控制工具

【反脆弱措施】
- 每完成一个独立变更（一个组件/一个功能点），立即 git add + git commit
- 需要回滚时，必须用 git revert / git reset，禁止手动编辑代码还原
- 如果不确定改动是否正确，先 commit 当前状态作为检查点，再继续尝试

#002 | 2026-05-12 | 越界修改：用户说移动图标，我却擅自缩小了尺寸

【问题】
用户要求将扫码图标从独立行上移到与"扫码归还"徽章同一行。
我在移动的同时，擅自将图标容器从 96px 缩小到 64px，图标从 48px 缩到 32px。
用户没有要求缩小，这属于超出需求范围的越界修改。

【根因】
- 自作主张认为同行放不下 96px 图标，"帮"用户做了布局判断
- 没有严格区分"用户明确要求的变更"和"我认为合理的调整"
- 遇到布局可能不够美观时，优先选择修改尺寸而不是忠实执行需求

【反脆弱措施】
- 严格执行"只做用户要求的事，不做用户没要求的事"
- 当觉得需要额外调整才能适配布局时，先忠实执行原需求，再单独询问是否需要微调
- 移动/重新排列元素时，保持原尺寸不变，除非用户明确指定新尺寸

#003 | 2026-05-12 | 理解偏差：把"同一行"理解为移到卡片外面

【问题】
用户要求"扫码卡片和扫码图标在同一行，但保持各自高度"。
我把图标移到了黄色卡片**外面**，和卡片并列。
用户实际意思是图标在黄色卡片**内部**右侧，和左侧内容同行。
连续两次都理解错了位置关系（内部 vs 外部）。

【根因】
- "同一行"的描述有歧义，我没有确认就按自己理解执行了
- 第一次放错后用户纠正，第二次又犯了同样的理解错误

【反脆弱措施】
- 遇到位置描述有歧义时（"同一行"、"并排"），先确认是在父容器内部还是外部
- 被纠正过一次的位置关系，必须作为明确的上下文记忆，不再重复犯错
